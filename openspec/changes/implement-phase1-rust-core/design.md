# Design: Phase 1 Rust Core

## Architecture

Phase 1 establishes a Tauri v2 application with a React + TypeScript frontend shell and a Rust backend core.

The frontend is intentionally minimal in this phase. It exists to satisfy the Tauri skeleton and should not contain conversion business logic.

The Rust backend owns:

- core mesh model.
- parser error model.
- `.fpn` keyword block parsing.
- conversion from keyword blocks into mesh data.
- parser tests.

## Expected Module Boundaries

Recommended Rust backend modules:

- `model`: node, element, element dimension/type, mesh data, group maps.
- `fpn_reader`: `.fpn` reading, keyword block collection, record parsing, group classification.
- `error`: parser and core errors.
- `lib`: backend module exports and Tauri app setup.
- `main`: Tauri binary entrypoint.

Exporter modules are intentionally not created in Phase 1 unless the Tauri skeleton requires placeholder wiring. If placeholders are needed, they must not implement exporter behavior.

## Data Model

The mesh model should preserve the C# data semantics:

- Node IDs and element IDs are signed 32-bit integers unless Rust APIs require a stricter alias.
- Coordinates are `f64`.
- Node and element maps use ordered maps to match C# `SortedList` iteration behavior.
- Element groups and node groups map group IDs to ordered ID lists.
- Group member order is the parsed order, not sorted after parsing.
- Element values store node IDs, not resolved node references.

The model should expose enough read access for future exporters without requiring the parser internals to leak.

## Parser Flow

The parser should follow the original C# flow closely:

1. Read the `.fpn` file as text.
2. Discover recognized keyword blocks.
3. Append continuation-line fields to the current block.
4. Parse nodes, elements, group names, and element groups.
5. Classify element groups by the first element ID.
6. Build `element_groups` from 3D, 2D, and 1D groups.
7. Build `element_group_names`.
8. Parse node groups using the original inclusion rule.
9. Build `node_group_names`.

Streaming may be used internally only if externally visible behavior remains compatible with the parser contract. The first implementation should favor clarity and compatibility over optimization.

## Parser Compatibility Rules

Phase 1 must preserve these documented behaviors:

- Keyword discovery is case-insensitive.
- Parsed keywords are trimmed before classification.
- Continuation lines are detected after trimming and must start with `,`.
- Element node IDs start at field index `2`.
- `MSETE` and `MSETN` IDs start at field index `8`.
- Default group ID `1` is ignored.
- Empty element groups are ignored.
- Mixed-dimension element groups are not detected or split.
- Node groups are included only if their group ID exists in `element_groups`.
- Ordered maps provide deterministic iteration.

## Error Model

Parser operations should return structured Rust errors rather than panicking.

The first error model must distinguish at least:

- input file not found.
- file read failure.
- malformed numeric value.
- duplicate ID.
- missing group name.
- malformed record shape.

The Tauri command error contract is defined separately in `docs/contracts/tauri-command-contract.md`. Phase 1 may add backend errors without fully wiring a conversion command.

## Tauri/Frontend Boundary

Phase 1 should create only the minimal frontend shell required by the Tauri application.

Frontend requirements:

- React + TypeScript.
- `npm` package management.
- no parser logic.
- no exporter logic.
- no full conversion UI unless generated by the skeleton and left minimal.

Backend requirements:

- Rust modules compile.
- parser tests run through `cargo test`.
- no exporter behavior is exposed.

## Test Strategy

Parser tests should use reference files:

- `GtsModel_Test.fpn`
- `GtsModel_Coarse.fpn`
- `GtsModel_Fine.fpn`

Required assertions:

- each reference file parses successfully.
- node count is nonzero.
- total element count is nonzero.
- parsed maps iterate deterministically.
- default group ID `1` is absent from parsed output groups.
- continuation-line records are handled without panic.

Tests should not assert exact output counts unless those counts are first confirmed and deliberately recorded. This avoids overfitting Phase 1 to accidental fixture details.

## Validation

Implementation validation should include:

- `cargo test` from the Rust backend location.
- frontend/Tauri validation command available after skeleton creation.
- `openspec validate implement-phase1-rust-core`.
